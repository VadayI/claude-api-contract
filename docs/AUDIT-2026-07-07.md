## 2. N1 — Витік глобального контексту через inline `@`-imports (P1, найбільший ефект)

**Механіка (верифіковано по офіційній доці “How Claude remembers your project”,
code.claude.com/docs/en/memory):**

1. `@path` у CLAUDE.md **імпортує файл у контекст при старті сесії** — і в import-блоці,
   і **посеред речення прози**.
2. Імпортовані файли **рекурсивно** імпортують свої `@`-посилання (до 4 hops).
3. `@path` у backticks або code-блоці **не** парситься як імпорт — це і є спосіб «згадати без імпорту».
4. «Splitting into @path imports does not reduce context» — розбиття на файли організує,
   але не економить: економить лише **відсутність** імпорту.

**Фактичний стан шаблону:**

- CLAUDE.md, крім 11 правил import-блоку, має **13 inline `@`-згадок у прозі**
  (рядки 24, 32–36, 48×2, 50, 66×2, 70, 74). З них нові глобальні імпорти:
  `typespec-style.md` (r.33), **`README.md` (350 рядків!)** + `node-commands.md` (r.66).
- 13 із 21 правил містять **27 крос-посилань `@.claude/rules/...` одне на одне**
  (найбільше: `typespec-style.md` — 5, `project-maturity.md` — 4). Через рекурсію ланцюги
  глобальних правил тягнуть майже всі скоуповані: `contract-first → auth-contract → api-envelope`;
  `verification → endpoint-surface → spectral-style`; `project-maturity → examples-validation → prism-mock`;
  `typespec-style → mcp-stack`, тощо.

**Наслідок:** дизайн «11 глобальних правил + 10 per-agent» (розділ *Rule scoping* CLAUDE.md)
**фактично не працює**: у кожну сесію CLI вантажиться ~20/21 правил (~700 рядків) + README (350)
≈ **1050 рядків замість задуманих ~350**. Скоупінг лишається тільки в описі.

**Фікс (механічний, застосовано в цій сесії):**

- CLAUDE.md: усі `@`-згадки в прозі → backtick-шляхи; import-блок (рядки, що починаються з `@`)
  недоторканий. Інструкція про `output-language.md` вже була в backticks — ок.
- `rules/*.md`: усі 27 крос-посилань → backtick-шляхи. Правила більше не імпортують правила;
  транзитивність і так зайва — агенти перелічують свої правила **явно** у власних тілах.
- `agents/*.md`, `commands/*.md`: `@`-посилання **збережені** — це і є задуманий механізм
  per-agent завантаження.

**Ефект:** глобальний контекст сесії ≈ −65% по шару правил; scoping-дизайн стає правдою.
Побічний бонус: `node-commands.md` («cited under Setup») і README справді стають citation, а не імпортом.

### 3.1 `engineering` — прибрати з baseline (рішення користувача; застосовано)

- Скіли **функціонально затіняють власних агентів**: `code-review` ↔ `contract-reviewer`,
  `system-design`/`architecture` ↔ `api-architect` + ADR-практика, `documentation` ↔ `docs-writer`,
  `testing-strategy`/`debug` — нерелевантні spec-only репо. Ризик: скіл тригериться замість
  доменного агента і обходить контрактні гейти (drift, envelope, x-surface).
- Команди `/standup`, `/incident`, `/deploy-checklist` — не мають сенсу в контрактному репо.
- Його `.mcp.json` постачає **другий GitHub MCP** (той самий endpoint, що й
  `github@claude-plugins-official`) → дубль інструментів у кожній сесії.
- **Дія:** видалено з `enabledPlugins` у `.claude/settings.json`. Derived-проєкти успадкують.

### 3.2 Потрійний GitHub MCP на робочій копії (локальний дефект; виправлено)

`settings.local.json` мав `enabledMcpjsonServers: ["github", "context7"]` — тобто fallback
`.mcp.json` було увімкнено **одночасно** з плагінами, всупереч власному правилу
(`mcp-stack.md`: «plugin або fallback — never both»). Разом з engineering виходило до
3× GitHub MCP. **Дія:** ключ прибрано (локальний файл, не в git). `.mcp.json` лишається
committed fallback'ом з `_note`.

## 4. N3 — Пари правило↔скіл: 6 дубляжів вмісту (P2, окремими PR)

Обраний принцип: **правило = норми** («що вимагаємо», завжди в контексті агента),
**скіл = рецепти** («як зробити»: CLI, синтаксис, приклади; вантажиться на вимогу).
Зараз обидва шари частково переказують одне одного і вже дрейфують.

| Пара | Перетин сьогодні | Що прибрати/лишити |
|---|---|---|
| `typespec-style.md` ↔ skill `typespec-authoring` | конвенції + синтаксис в обох | правило: layout/naming/semver-інваріанти; скіл: декоратори, приклади, emitter-опції |
| `spectral-style.md` ↔ skill `spectral-lint` | список вимог + рецепти правил | правило: ЩО enforce'иться (казуси, x-surface, envelopes); скіл: синтаксис rules, functions, приклади |
| `prism-mock.md` (rule) ↔ skill `prism-mock` | режими/прапорці в обох | правило: інваріанти (`-h 0.0.0.0`, `-m false`, two-way validation як гейт); скіл: команди запуску, x-faker рецепти |
| `breaking-changes.md` ↔ skill `oasdiff-breaking` | класифікація + CLI в обох | правило: ERR→major, ignore-файл→ADR; скіл: oasdiff CLI, changelog рецепти |
| `versioning.md` ↔ skill `contract-versioning` | semver-політика + release-флоу | правило: політика й consumer-пінінг; скіл: покрокові release/pin рецепти |
| `api-envelope`+`auth-contract`+`endpoint-surface` ↔ skill `openapi-design` | скіл переказує конспект трьох правил | скіл → «індекс + REST-евристики», конкретика лишається в правилах |

Обсяг: 6 малих PR (по парі), кожен ≤2 файли. Критерій готовності: жоден факт не живе
у двох місцях; перехресні згадки — **backtick-шляхами** (див. N1).

## 6. N5 — Family-core plugin: дизайн + план міграції (обраний напрямок дистрибуції)

**Проблема:** три шаблони (`claude-api-contract`, `claude-django`, `claude-react-mui`)
триплікують спільне ядро: агенти `ba`, `docs-writer`, `devil`, `auditor`, `brief-synthesizer`,
`template-sync`; команди `/audit`, `/doctor`, `/preflight`, `/bootstrap`, `/handoff`,
`/wrap-up`, `/synthesize-brief`, `/set-language`, `/fix-ci`, `/review-pr`, `/create-pr`,
`/verify`, `/plugins`, `/update-from-template`; правила `workflow`, `no-stubs`,
`git-operations`, `preflight`, `environment`, `mcp-stack`, `living-plan`, `verification`,
`node-commands`, `project-maturity`. Кожен фікс ядра = 3 PR у 3 репо. Водночас користувач
хоче «додавати команди/агентів/скіли до вже існуючих проєктів» — це буквально модель плагіна.

**Ключове обмеження механізму плагінів:** плагін постачає commands / agents / skills /
hooks / MCP — але **не** `rules/` і не CLAUDE.md. Тому:

- **У плагін** їдуть агенти + команди + скіли + спільні скрипти (hook-скрипти можна везти
  в плагіні, посилаючись через `${CLAUDE_PLUGIN_ROOT}`).
- **У репо** лишаються правила (тонкі, доменні) + CLAUDE.md + гейт-скрипти CI.
- Плагінові агенти **не можуть** `@`-імпортувати репо-правила відносним шляхом — але їм і
  не треба: субагент і так отримує project-context (CLAUDE.md + його import-блок).
  Норми ядра пишуться **самодостатньо в тілі агента**; доменну специфіку агент читає з
  CLAUDE.md/PROJECT.md проєкту, в якому запущений.

**Архітектура:**

```
github.com/VadayI/claude-family-marketplace      <- marketplace (semver-теги)
  plugins/family-core/
    agents/    ba.md devil.md auditor.md brief-synthesizer.md template-sync.md docs-writer-core.md
    commands/  audit.md handoff.md wrap-up.md set-language.md synthesize-brief.md update-from-template.md
    skills/    project-maturity/ living-plan/ preflight-core/   (норми, що були б «правилами», їдуть скілами)
    scripts/   log-cmd.mjs detect-env.mjs session-start.sh
    hooks/     hooks.json (SessionStart -> ${CLAUDE_PLUGIN_ROOT}/scripts/session-start.sh)
```

**Правило розшарування:** якщо текст однаковий у 3 репо — ядро (плагін); якщо містить
доменні слова (contract/spec/openapi · django/DRF · react/MUI) — лишається в репо.
Приклад: `workflow.md` НЕ їде в ядро (пайплайни різні); `no-stubs.md`, `git-operations.md`
їдуть як скіли/тіла агентів ядра, репо тримає 3-рядкові доменні доповнення.

**План міграції:**

1. **Phase 0 (ADR 0011):** зафіксувати рішення, межу ядро/домен, semver-політику плагіна,
   fallback (репо працює і БЕЗ плагіна — деградація, не поломка).
2. **Phase 1 (низький ризик):** marketplace-репо + `family-core v0.1.0`: `auditor`,
   `template-sync`, `/audit`, `/handoff`, `/wrap-up`, `/set-language`, `log-cmd.mjs`.
   Пілот у `claude-api-contract`: додати в `enabledPlugins`, видалити локальні дублікати,
   тиждень експлуатації.
3. **Phase 2:** `ba`, `devil`, `brief-synthesizer`, `/synthesize-brief`, session-start/detect-env
   (обережно: у них доменні хвости — виносити generic-частину).
4. **Phase 3:** розкатка на `claude-django` + `claude-react-mui` (їхні локальні копії → delete,
   доменні дельти → у їхні rules). `/update-from-template` + `template-sync` звужуються до
   доменного шару; оновлення ядра = bump тега плагіна.
5. **Не-цілі:** гейт-скрипти CI (`check_*.sh`) лишаються в репо — вони запускаються без
   Claude; правила-домени лишаються правилами.

**Ризики:** дрейф версій плагіна між репо (мітигація: `/doctor` перевіряє мін. версію ядра);
поведінкові відмінності Cowork vs CLI для плагінових hooks (перевірити в Phase 1);
«скіл замість правила» тригериться не гарантовано (тому інваріанти безпеки — тільки
в тілах агентів або репо-правилах, не в скілах ядра).
