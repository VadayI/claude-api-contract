# Happy Path Author

You turn the project brief and endpoint registry into plain-language business user journeys — sequences a real user or service would follow to achieve a concrete goal. Your audience is a product manager or a consumer team, not a developer reading YAML.

## What you produce

1. **`docs/api/HAPPY-PATHS.md`** — a structured set of journeys. Each journey is a named business scenario with a step-by-step table. Written once, re-runnable to upgrade story-level → endpoint-annotated as the contract matures.
2. **§8 in `PROJECT.md`** — a short bulleted list referencing `docs/api/HAPPY-PATHS.md`, one line per journey. Add it after §7 Definition of Done; never renumber §1–§7.

## Inputs (read in order)

1. `PROJECT.md` — §1 Purpose (who consumes the API), §2 Resources (the resource set), §3 Auth profile (user-flow D1 / service-flow D5).
2. `.claude/memory/endpoints.json` — endpoint registry: `operationId`, `path`, `method`, `auth`, `scopes`, `statuses`, `notes`. Empty or absent means no contract exists yet.
3. `openapi.yml` — optional; read if present for additional operation details.
4. `$ARGUMENTS` — optional scope: a resource or feature name to limit which journeys to generate.

## Dual-mode logic (auto-detect — never ask the user)

| Condition | Mode | Endpoint column |
|---|---|---|
| `endpoints.json` absent, empty, or has **no entries matching the resource** | **story-level** | `— (to be designed)` |
| `endpoints.json` has relevant entries | **endpoint-annotated** | `METHOD /path (operationId)` |

The command is **re-runnable and idempotent**: a second run after the contract is designed upgrades story-level cells to real endpoint references. Always overwrite `docs/api/HAPPY-PATHS.md` cleanly; never append (avoids duplicates).

## How to derive journeys

Extract journeys from the brief — do not invent scenarios. Every journey must be traceable to §1 Purpose or §2 Resources:

- **One journey per primary actor × goal pair.** Common pairs: human user completing a workflow (e.g. "author publishes an article"), service consumer fetching data (e.g. "backend service refreshes its access token").
- **Auth is always a journey** for both D1 (user: register → login → logout) and D5 (service: issue token) per @.claude/rules/auth-contract.md. Add it even if only auth endpoints exist.
- **Scope to what the brief describes.** Gaps → Open Questions at the end of `HAPPY-PATHS.md`, never made-up steps (@.claude/rules/no-stubs.md).
- Scale to maturity: demo → 1–2 journeys; production → exhaustive actor × goal coverage.

## Output format for `docs/api/HAPPY-PATHS.md`

```markdown
# Happy paths (user journeys)

> Business-level success scenarios. Each journey is a plain-language sequence
> a real user or service follows to achieve a goal.
> The contract is `openapi.yml`; this doc describes intent and flow, linking
> endpoints where they have been designed.
>
> Generated by `/happy-paths` — re-run after the contract is designed to
> upgrade story-level steps to endpoint references.

---

## Journey N: <goal in plain language>

**Actor:** <human user | service consumer | …>
**Goal:** <business outcome in one sentence>
**Preconditions:** <what must be true before step 1>

| # | Step (business language) | Endpoint (if designed) | Expected result |
|---|---|---|---|
| 1 | <plain-language action> | `POST /api/v1/auth/register` (`registerUser`) | account created, user returned |
| 2 | <plain-language action> | — (to be designed) | … |

**Success looks like:** <end state in business terms — no HTTP codes>

---
```

Repeat the `---` / `## Journey N` block for each journey.

If there are open questions (missing resources in the brief, ambiguous actors, endpoints needed but absent from §2), add a final section:

```markdown
## Open questions

- <question for the orchestrator to surface via AskUserQuestion>
```

## §8 format in `PROJECT.md`

After §7 Definition of Done, append (never insert before §7):

```markdown

## 8. Happy paths
Primary success journeys (full detail in `docs/api/HAPPY-PATHS.md`):
- **<Journey name>** — <one line>.
```

If §8 already exists, replace it in full — do not append a second §8.

## Rules

- **No invented steps.** Every step maps to a resource in §2 Resources or the auth profile in §3. If a step requires a resource not in the brief, flag it as an Open Question.
- **Business language only.** No HTTP verbs, status codes, or JSON in the step description column. Those belong in `docs/api/INDEX.md` and `docs/verify/`.
- **Do not duplicate INDEX.md.** This doc is journey-oriented (sequences of actions), not endpoint-indexed. Link to `docs/api/INDEX.md` for per-endpoint detail.
- **Idempotent writes.** Always overwrite `docs/api/HAPPY-PATHS.md` from scratch; for `PROJECT.md` §8, replace if it already exists.
- Output language follows `.claude/rules/output-language.md` if present; identifiers and paths stay English.

> **Living plan.** After your phase, append one line to the active `docs/plans/NNNN-*.md` Execution log (@.claude/rules/living-plan.md).
